Below is an independent review of the draft as a PODS submission, based only on the text you pasted.

Overall assessment

This is a promising and potentially important paper, but in its current form I would worry about it being rejected at PODS for overclaiming relative to the formal model and for a few definition/proof mismatches that make the main theorem feel tautological rather than substantive.

The core idea is strong:

Move CALM from program syntax / set inclusion to semantic specifications over Lamport histories and arbitrary outcome-refinement orders.

That is a good paper idea. It is relevant to PODS, especially if framed as a semantic unification of CALM, HAT/I-confluence, CRDTs, and ordered-output systems.

But the current draft frequently claims more than it proves. The biggest danger is that a reviewer says:

“The theorem is true because coordination-free was defined to mean exactly monotonicity plus no suppression. The hard work has been pushed into the definitions, and several applications are only sketches or are technically incorrect.”

I think the paper can be repaired, but it needs tightening.

Executive summary of main concerns

1. The main theorem is too tautological as currently stated

The proof of Complete CALM is almost immediate because:

* coordination-free requires Expose_I(H) = Obs(H);
* implementations are required to be irretractable;
* irretractability is exactly monotonicity over exposed outcomes;
* therefore coordination-freedom iff Obs is monotone.

That is not necessarily bad. CALM itself has simple proofs once the definitions are right. But here the draft repeatedly says things like “universal criterion,” “no further generalization possible,” and “exact semantic characterization,” while the definitions make the theorem almost definitional.

The paper needs to acknowledge this more directly and make the contribution be:

the modeling theorem: the right semantic abstraction of coordination-freedom is no history suppression + no outcome suppression + irretractable observations, and under that abstraction monotonicity is exactly the boundary.

Right now it sounds like a hard theorem, and reviewers will punish that.

2. The implementation model is underdefined and possibly inconsistent

Several definitions do not quite line up.

The role of H_in

Implementation defines R_I(H_in) for each input history, but Expose_I : Hist -> P(O) is global. Correctness says:

Expose_I(H) ⊆ Obs(H) for all H ∈ R_I(H_in).

But a history can belong to many R_I(H_in) sets, and the dependence on H_in mostly disappears. In the proof of necessity, you say:

By condition (i), R_I = A, so both H_1 and H_2 are realized.

But R_I is indexed by H_in. For arbitrary H_1 \hext H_2, which H_in are you using? Presumably H_in = In(H_1) or the least input prefix shared by both. This needs to be explicit.

A cleaner version might define an implementation as a subset R_I ⊆ Hist downward-closed, plus optional liveness/input-admissibility constraints, rather than indexing by input prefix unless that indexing is essential.

“No history suppression” is too strong and misdescribed

You write:

Condition (i) says the implementation never prevents a causally admissible future from occurring—it cannot block message deliveries, delay inputs, or otherwise constrain the set of reachable histories.

But real coordination-free systems absolutely do not realize all possible internal histories. They have specific algorithms. They may not send certain messages, may not perform certain internal steps, etc. What they should not do is require cross-node coordination to decide whether a locally enabled external observation is safe.

As written, R_I = A seems to require realizing all histories containing arbitrary internal events and message sends/receives in the event universe. Unless E_int, E_send, and E_recv are already restricted to the implementation’s possible events, this is impossible.

This issue matters because the main theorem quantifies over all well-formed histories, not over histories generated by a transition system. You need to clarify whether Hist is:

1. the specification’s physically possible histories,
2. the implementation’s operational histories,
3. the environment’s admissible histories,
4. or some product of specification + implementation + network.

Right now the framework says “specification fixes event universe,” but then implementations choose histories. That makes “realize all admissible histories” too blunt.

3. The running linearizability example appears technically wrong

The key witness for non-monotonicity of linearizability is not convincing as written.

At H_1, the read has an invocation but no response. You say both:

\langle w(1), r \mapsto 0 \rangle
\quad\text{and}\quad
\langle w(1), r \mapsto 1 \rangle

are admissible completions under linearizability.

But if outcomes are “sequences of completed operations annotated with return values,” then at H_1 the read is not complete. The outcome should probably be just:

\langle w(1) \rangle

unless Obs(H) includes possible completions of pending operations. The draft oscillates between:

* outcomes that have actually completed by H; and
* completions that are possible in some future extension of H.

That distinction is crucial.

Then at H_2, you say:

the write’s response precedes the read’s response in real time, so every linearization places w before r.

For linearizability, real-time order is usually: if operation A’s response precedes operation B’s invocation, then A must precede B. A write response preceding a read response is not enough. In your H_1, the read invocation is concurrent with the write response, so even after the read returns 1, the operations overlap. Linearizability would still allow the read to be linearized before the write, unless the semantics of the returned value rules that out. If the read returns 1, then yes it must see the write; but if the read had returned 0, the overlap may still allow that.

So this witness does not show that r -> 0 becomes invalid solely because the message arrived before the response. Under standard linearizability, a concurrent read that started before the write completed can return either old or new value.

To get a clean witness, use real-time order:

* H_1: write invocation exists but is incomplete; read invocation/response completes returning 0; this is admissible because the write has not completed.
* H_2: extend by adding the write response before the read invocation? But futures cannot insert predecessors of existing events, so that will not work.
* Therefore the future relation itself makes standard real-time-order witnesses tricky.

This reveals a deeper issue: with Lamport futures that cannot insert new predecessors, many standard linearizability facts are prefix-stable once an operation has completed. Linearizability may be non-monotone with respect to choosing a completion of pending operations, but not necessarily with respect to completed-operation observations under prefix extension.

You need to decide whether Obs(H) is:

1. observations already justified by completed events in H; or
2. possible future completions of pending operations extending H.

If it is (2), then the theorem is about speculative exposure of possible completions, not ordinary observable outputs. That can work, but it must be stated carefully.

4. “Eventual consistency is monotone” is not sufficiently specified

The eventual consistency spec says a completed read may return the initial value or any value written in the causal past. But then outcomes are sequences ordered by prefix extension. If a read’s return value is included in the sequence, a future cannot change it. So monotonicity depends on whether future histories can add constraints that invalidate that return. For the EC spec as written, probably yes, but only because the spec is permissive.

However, “eventual consistency” usually includes eventual visibility under delivery/quiescence assumptions. You explicitly assume no fairness or delivery guarantees. Then the spec is closer to “weak/causal-ish stale-read permissiveness” than eventual consistency. A PODS reviewer may object.

Use a more precise name, e.g.:

* Spec_stale
* “stale-read register”
* “available register with arbitrary staleness”
* “causally permissible stale reads”

Then discuss eventual convergence separately if needed.

5. The HAT / isolation section has likely correctness problems

The serializability witness is not fully right.

You define outcomes as sets of facts with set inclusion. At H_1, T1 commits after reading x=0. At H_2, T2 also commits reading y=0. You claim the two reads induce a cyclic serialization order.

Let initial values be x=y=0.

* T1: reads x=0, writes y:=1.
* T2: reads y=0, writes x:=1.

If both commit and both read 0, then yes this is the classic write-skew / anti-dependency cycle under serializability.

But serializability itself does not necessarily make read facts monotone under set inclusion unless your outcome includes the full committed transaction set and read-from constraints. You need to specify whether Obs_ser(H_1) at a partial history can include T1 alone even if a concurrent T2 later commits. In ordinary serializability, a prefix with only T1 committed can be extended by aborting T2; but if T2 later commits with incompatible read, the earlier “commit T1” fact alone is not contradicted. What is contradicted is the combination “commit T1 with read x=0” plus “commit T2 with read y=0.” But under set inclusion, o1 can be extended by an o2 that includes commit(T1) and omits commit(T2) unless H_2 forces committed transactions to appear in every outcome.

So you need a stronger definition of Obs_L(H):

at a history containing a commit decision, every observable outcome must include that committed transaction and its read/write facts.

Without that, non-monotonicity does not follow.

Also, the statement:

The same argument applies to snapshot isolation (write-skew creates the cycle). By Complete CALM, serializability and snapshot isolation intrinsically require coordination.

This is risky. Snapshot isolation is not serializability; write skew is allowed by SI. SI may require coordination for write-write conflict detection / first-committer-wins, depending on model, but “write-skew creates the cycle” is not the right reason SI itself is non-monotone. If anything, write skew is the classic anomaly SI permits. This will trigger database reviewers.

I would either remove SI from the claim or give a separate, precise SI witness based on write-write conflict / commit ordering.

6. The I-confluence section violates your own nonemptiness condition

You define specifications as requiring Obs(H) ≠ ∅ for all histories. Then in I-confluence you set:

\[
\Obs_I(H) = \emptyset
\]

when the converged state violates the invariant, and say you “formally relax” the earlier requirement.

That is not a small local relaxation. It changes the main theorem. The theorem assumes nonempty observations; if bad histories have empty observation sets, then non-monotonicity can be witnessed by emptiness, but coordination-freedom and correctness need to be restated for partial specs / safety specs / forbidden histories.

This is repairable, but it needs to be elevated into the core model. Probably introduce:

* total specs: Obs(H) ≠ ∅;
* safety specs: Obs(H) may be empty, meaning H is forbidden;
* coordination-free implementation must realize all admissible histories whose inputs are legal, or must suppress histories with empty Obs?

But if coordination-free requires no history suppression, no spec with any empty Obs(H) can be implemented coordination-free. That might be exactly what you want for non-I-confluent invariants, but the model needs to say so.

Also the proof direction labels are swapped:

(\Rightarrow) Assume T is I-confluent.
...
(\Leftarrow) Assume T is not I-confluent.

The proposition says Spec_I is monotone iff T is I-confluent. The first direction should be (I-confluence ⇒ monotonicity), not (\Rightarrow) unless the left side is I-confluence. As written, it is confusing.

7. The “Subsumption of CALM” theorem has a serious mismatch

Definition Transducer instantiation says:

Histories are prefixes of fair runs of the transducer network.

But earlier Hist is all well-formed Lamport histories over an event universe. If here histories are only prefixes of runs of a particular transducer network, then the specification already includes the implementation. That collapses specification and program.

Also:

\[
\Obs(H)=\{O_H\}
\]

where O_H is the set of output facts irretractably produced by history H.

Then theorem says this is equivalent to monotonicity of query Q.

But if outputs are irretractable by construction, then for any run prefix H_1 \hext H_2, O_{H_1} ⊆ O_{H_2} regardless of whether Q is monotone. A non-monotone query can be implemented by a transducer that delays output until enough coordination has occurred; its emitted outputs are still irretractable. So Spec_Q as defined is monotone because it describes emitted output of a particular run, not semantic query output over inputs.

This is a big issue.

To recover CALM, Obs(H) should probably be semantic query outputs determined by the input facts visible/available at H, not the outputs already produced by a particular transducer. Or you need to distinguish:

* program-output monotonicity along runs, which is trivial under irretractable output;
* query monotonicity with respect to input growth, which is CALM’s content.

As written, the proof of (3) => (2) says:

Since outputs are irretractable, O_H1 ⊆ O_H2.

That does not use monotonicity of Q. That is a red flag: the theorem is not proving what it claims.

8. The “separation from CALM” theorem overstates CALM’s limitations

The separation idea is valuable: coordination can produce a monotone artifact, such as a log, and downstream consumers can be coordination-free.

But the theorem says:

CALM, applied syntactically to any program implementing I, classifies it as non-monotone.

This is too strong. A program implementing a log consumer can be monotone. A program implementing Paxos is non-monotone. But “any program implementing I” depends on what the program’s input/output boundary is.

Also, CALM was never intended to say that consumers of an append-only log require coordination just because the producer used coordination. You can express append-only log consumption as monotone Datalog over growing log facts. So the distinction may not be “CALM cannot make this distinction,” but:

program-level CALM gives answers relative to a chosen program boundary; Complete CALM gives a semantic way to choose and verify the boundary at the level of observable outcomes.

That is subtler and more defensible.

The stratified Datalog example has the same problem. After sealing, the final unreachable relation is monotone only because the input domain is closed. But from CALM’s perspective, this is just changing the input model: if edge is sealed and no longer grows, then unreachable can be emitted monotonically during the second phase. This is not obviously outside CALM; it is a staged/stratified coordination story.

I would reframe the separation as:

Complete CALM is boundary-sensitive and semantic: it can certify the residual interface after a coordination mechanism. Classic CALM, as normally applied to a monolithic program over growing input facts, does not express this residualization without explicitly modeling sealing/logging as part of the input semantics.

That avoids an unnecessarily antagonistic and probably challengeable claim.

9. Arbitrary outcome orders are powerful, but the paper underplays “cheating” risk

The draft has a good remark about too-coarse orders. But reviewers will still ask:

If I can choose any partial order, can I make any spec monotone?

You say a top element refining everything trivializes monotonicity, but then rely on “semantic contract” to rule that out. That may be too informal for PODS.

You need a stronger discipline, perhaps:

* outcome orders must be faithful to an observational equivalence / client-visible compatibility relation;
* if two outcomes cannot both be valid observations in a single execution, they must have no common upper bound;
* refinement must preserve all already-observed client-visible facts;
* the order should be part of the externally specified object, not an analysis artifact.

This should be formalized, not just remarked.

A useful definition might be:

A refinement order is faithful if whenever o1 ⪯ o2, every client observation entailed by o1 is also entailed by o2.

Then the theorem is parameterized by faithful orders. Without this, the “universal criterion” claim is vulnerable.

10. The universal construction is currently too broad / false as phrased

The theorem says:

For any specification, membership + total ordering yields a monotone residual specification.

But the proof changes the outcome interface:

“Note that this defines a new residual output interface … not the original specification’s outcome domain.”

That is not just residualization; it is a different specification. If arbitrary interface change is allowed, almost anything can be made monotone by exposing a log of decisions. This undercuts the main theorem.

The theorem should be demoted or reframed as an architectural corollary:

Any implementation can expose its input log monotonically; deterministic consumers of that log are coordination-free with respect to the log interface.

But it does not show that any original specification can be made monotone after ordering unless the original outputs themselves are prefix-refinable / emitted irretractably.

Also this paragraph is suspicious:

The universal construction resolves all distributed coordination in one round (membership).

But total-order broadcast / consensus per entry is ongoing distributed coordination. Later you acknowledge that, but the “one round” line in the remark conflicts with it. This will confuse or annoy reviewers.

What I think the paper should become

The best version of this paper is not “we prove a difficult new theorem.” It is:

We identify the semantic object that CALM was always secretly about: monotone growth of client-visible commitments over causal futures. This gives a language-independent criterion and explains how coordination mechanisms transform non-monotone specs into monotone residual interfaces.

That is a very nice paper. But to make it credible at PODS, I would restructure around three increasingly strong claims:

Claim A: Semantic CALM

Define specs over histories, observations, and faithful refinement orders. Prove:

A spec is safely exposable without suppressing futures iff observations are future-monotone.

This is the main theorem.

Claim B: Recovery of known results

Show carefully, with correct instantiations:

* relational transducer CALM;
* HAT boundary;
* I-confluence;
* CRDTs.

But these must be exact or clearly labeled as sketches.

Claim C: Residualization / well-coordination

Define coordination mechanisms as restrictions/transforms of a spec. Prove:

If a coordination mechanism produces a monotone residual interface, then downstream consumers are coordination-free.

This is the compositional contribution. It should not be framed as “CALM cannot do this,” but as “classic CALM lacks this semantic residualization operator.”

Concrete fixes by section

Title

“Complete CALM” is catchy, but dangerous. It invites a reviewer to say it is not complete, because:

* arbitrary outcome orders need faithfulness constraints;
* progress/liveness are excluded;
* partial specs with empty observations are handled ad hoc;
* coordination-freedom is defined semantically rather than operationally.

Possible safer titles:

1. Semantic CALM: Coordination-Freedom for Ordered Observations
2. Future-Monotonicity and Coordination-Freedom
3. A Semantic CALM Theorem for Distributed Specifications
4. Coordination as Future-Inconsistent Observation
5. Residual CALM: Reasoning About Coordination and Its Outputs

I would avoid “Universal Criterion” unless the model is made much more precise.

Abstract

The abstract overclaims:

“We recover the HAT/non-HAT boundary, invariant confluence, and CRDT coordination-freedom as instances…”

This is fine only if those sections are formal and correct. Currently they are sketches.

I would soften:

“We show how the framework recovers or clarifies…”

Also:

“classical ordered properties … such as sequential consistency”

Sequential consistency itself requires coordination. What you mean is that resolved ordered logs are monotone under prefix extension. Say that.

Suggested abstract sentence:

“The framework separates the coordination needed to choose an order from the coordination-free consumption of the resulting ordered prefix.”

That is clearer.

Introduction

The introduction is compelling, but it should more explicitly distinguish:

* original spec;
* coordination mechanism;
* residual/output spec.

Right now the paper sometimes says linearizability is non-monotone and sometimes says linearizability resolved via Paxos is monotone. That is true only because the observable interface changes from “operation returns under concurrency” to “prefix of decided log.”

Add a paragraph early:

We distinguish three objects: the unconstrained specification, which may be non-monotone; a coordination mechanism, which restricts histories or outcomes; and the residual specification exposed after coordination. Our theorem applies to each. A non-monotone spec requires coordination; a successful coordination mechanism produces a monotone residual interface.

That would make the whole paper easier to read.

Running example

I would replace the current linearizability witness. It is too fragile.

A cleaner example may be a single-assignment register / unique name allocation / first-writer-wins spec.

Example:

* Two nodes concurrently propose values a and b.
* At history H1, only proposal a is visible to node p, so outcome choose(a) is admissible.
* At future H2, proposal b with higher priority / earlier timestamp becomes visible and the spec requires choose(b).
* Then choose(a) has no refinement.

But be careful with futures: if b must be causally earlier than an existing event, you cannot add it later. So use a spec where b is concurrent but not previously known, and its later delivery changes admissibility. For example:

* H1 contains both proposals as input events? If so, the node cannot see both? Your histories are global, so visibility is not local.
* You may need local histories / per-process observations, or histories that include unknown concurrent events? Lamport histories as global prefixes do not support “later discovering an already-concurrent event” unless the event was not in the prefix.

This is important: your future relation only adds causally later events. It cannot add an event concurrent with existing events if that event has no causal relation to existing events? Actually it can add a new event e' concurrent with all existing events; condition (iii) only forbids e' -> e for existing e. So a future can add a concurrent input event. That gives you a clean witness.

Use something like:

* At H1: input choose(a) at p has occurred; outcome a is admissible.
* At H2: add concurrent input choose(b) at q; spec requires a deterministic winner by global minimum proposal id, and b<a; now only b is admissible.
* Outcome a is stranded.

This shows non-monotonicity of total-choice specs under addition of concurrent events. It is much cleaner than linearizability.

Then later discuss linearizability as an application with more care.

Framework

I would revise the definitions as follows.

Add local observation / visibility or drop “local information” claims

The model currently uses global histories. Then it says:

implementation can respond using only locally available information

But nothing in the formalism defines local views. If the theorem is about global histories and semantic suppression, avoid claiming it models local information directly.

Either add local views:

view_p(H)

and define coordination-free exposure at process p as depending only on view_p(H), or remove local-information language.

Classic CALM is about independent nodes computing without coordination; if you do not model local views, reviewers may say you characterize irretractable global observability, not distributed coordination.

Clarify Obs

Is Obs(H):

* what may be emitted now at H;
* what may eventually be emitted in some extension;
* or all outcomes consistent with H?

The theorem only works if Obs(H) means “safe-to-expose now.” But examples often use “admissible completions.” Decide and enforce.

I recommend:

Obs(H) is the set of outcomes that the specification permits an implementation to expose at history H.

Then pending-operation completions should not be in Obs(H) unless the spec permits speculative exposure.

Move irretractability into observation semantics?

Right now implementations are required to be irretractable. Then coordination-free requires exposing all Obs. Thus monotonicity follows.

You might instead define:

* safe exposure of o at H means future-monotone from H;
* coordination-free means every admissible observation is safe.

That makes the theorem feel less circular.

Main theorem

The theorem should be preceded by a sentence like:

The proof is immediate once the definitions are in place; the substance lies in choosing the semantic objects so that the theorem applies across models.

This disarms the “tautology” criticism.

But then the applications must carry the weight. They need to be made more formal.

Subsumption of CALM

This section needs the most repair.

The current instantiation should not define Obs(H) as already produced outputs of the transducer. Instead, perhaps:

* histories contain input arrivals;
* outcome is the query result on the set of input facts present in the history;
* order is set inclusion.

Then monotonicity of Spec_Q is exactly monotonicity of Q.

But to recover Ameloot’s coordination-freedom, you need more than this. You need to map their operational model to your “no suppression” model. That may be a theorem but should be stated carefully.

Suggested simpler version:

For relational queries over growing input instances, take histories to be input-fact growth, outcomes to be query results, and refinement to be set inclusion. Then future-monotonicity of Spec_Q is exactly query monotonicity. By the Ameloot et al. CALM theorem, this coincides with coordination-free transducer evaluation.

That is enough. You do not need to prove equivalence of definitions in full unless you are prepared to formalize the transducer model.

Beyond set inclusion

The point is good, but the current sequential consistency discussion is muddled.

Say:

Set inclusion is the wrong order for ordered streams. Prefix order is the natural refinement relation. Encoding a stream as a set of position facts loses the semantic invariant that there is a single value per position; enforcing that invariant reintroduces non-monotone constraints.

But do not say:

“The classification is wrong: the specification is coordination-free under its natural order.”

Sequential consistency / linearizability are not coordination-free before the order is chosen. The resolved log is coordination-free to consume. Use that distinction rigorously.

Separation / well-coordination

This is potentially the most novel part. Make it central and precise.

I would define a residualization operator:

Given a coordination mechanism I, define:

\[
\Spec/I = (E_I, \Obs_I, \Ord_I)
\]

But be careful: if Ord_I or outcome domain changes, this is not a restriction of the original spec. It is an abstraction/interface exposed by the mechanism.

Maybe distinguish:

1. Restriction residual: same outcome domain/order, fewer histories/outcomes.
2. Interface residual: new outcome domain/order, produced by an observation map from mechanism states.

The Paxos log example is an interface residual, not just a restriction.

This distinction would save the universal construction too.

Applications

HAT

Use Bailis et al. carefully. The HAT paper’s boundary is not exactly “RU, RC, session guarantees monotone; serializability and SI non-monotone.” It classifies several guarantees under high availability, sticky availability, transactional availability, etc. Be precise.

I would narrow the claim:

We illustrate the mapping on representative HAT and non-HAT guarantees.

Do not claim full recovery unless you prove every case.

I-confluence

Move partial specs / empty Obs into the core framework before this section.

Also: uniqueness under merge strategies is tricky. The statement:

“foreign keys are not I-confluent … while uniqueness under certain merge strategies is”

needs care. In the original I-confluence results, uniqueness constraints usually require coordination under common set-union semantics because two replicas can independently insert the same unique key with different records. Some uniqueness variants become confluent only with escrow/allocation or deterministic conflict resolution. Say that explicitly or avoid the example.

CRDTs

This is the strongest application section. It should maybe appear first among applications because it is clean.

But the proof statement:

every state reachable at H' is at least as large as some state reachable at H

is not enough for the required quantifier:

For every o ∈ Obs(H), there exists o' ∈ Obs(H') with o ⊑ o'.

If Obs(H) contains multiple possible states due to nondeterministic application order, you need to show each one can be extended along the same future. For state-based CRDTs with join/inflationary updates, that is probably fine, but the proof should say:

take the execution producing o at H; append the additional updates/merges from H'; inflationarity gives the extension.

Also, reset-capable CRDTs are subtle. Some reset/remove operations can be implemented as CRDTs by enriching the lattice with epochs/tombstones/version vectors. The point should be:

reset is non-monotone with respect to the plain numeric order; implementing it coordination-free requires changing the state space/order so that the reset is represented inflationarily.

That is exactly aligned with your theory.

Suggested revised theorem structure

Here is a more defensible core.

Definition: specification

A spec is:

\[
(E, O, \preceq, \Poss, \Obs)
\]

where:

* Poss(H) are outcomes still semantically possible after history H;
* Obs(H) ⊆ Poss(H) are outcomes allowed to be exposed at H;
* o ⪯ o' means o' preserves/refines the client-visible commitments of o.

Then define future-monotonicity of exposed observations:

\[
\forall H \hext H',\ \forall o \in \Obs(H),\ \exists o' \in \Obs(H') : o \preceq o'.
\]

This lets you distinguish:

* possible completions;
* observable commitments;
* safe exposure.

Definition: coordination-free

An implementation is coordination-free if it does not restrict admissible futures and exposes exactly the spec’s allowed observations.

But perhaps avoid saying it must expose “every” outcome. Say instead:

every outcome in Obs(H) is independently safe to expose.

Then Expose_I(H) is a set of safe choices, not simultaneous outputs.

Theorem

A spec is coordination-free exposable iff Obs is future-monotone.

Then the proof is clean and honest.

Bottom-line recommendation

The draft has a strong conceptual center, but it currently reads like it is trying to prove too much too fast. For PODS, I would make it narrower, sharper, and more formally disciplined.

The winning paper is probably:

Semantic CALM: future-monotone observations over faithful refinement orders exactly characterize coordination-free exposure; coordination mechanisms can be understood as residualizing non-monotone specs into monotone interfaces.

That is elegant, defensible, and useful.

The current version’s riskiest claims are:

* “Complete CALM” as a universal final theorem;
* exact formal subsumption of CALM as currently defined;
* the linearizability witness;
* the HAT/SI and I-confluence applications;
* the assertion that CALM cannot recognize monotone residual outputs.

I would not submit this version unchanged. I would revise toward a more careful semantic paper with fewer but stronger claims.
